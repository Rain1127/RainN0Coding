# 本轮修复问题方案沉淀

本文档记录本轮代码治理中已经完成落地的修复项，统一按“问题 + 方案”的格式沉淀，便于后续复盘、交接和继续治理。

## 1. Python 工作流导入失败

**问题**

`python-agent/agents/builder_agent.py` 中缺少可导出的 `builder_agent` 函数，导致 `workflow/code_gen_workflow.py` 在导入时触发 `ImportError`，整个代码生成工作流无法正常组装。

**方案**

在 `builder_agent.py` 中补回明确的 `builder_agent(state: CodeGenState) -> CodeGenState` 入口，将原有 Builder 主流程收敛到该函数中，恢复工作流对 Builder 节点的标准调用方式。

同时补充回归测试，直接校验工作流模块可被导入并正确构建。

**落地结果**

Python 工作流导入恢复正常，`create_code_gen_workflow` 可以成功创建流程对象。

**涉及文件**

- `python-agent/agents/builder_agent.py`
- `python-agent/tests/test_workflow_imports_unittest.py`

## 2. Python 文件工具存在路径穿越风险

**问题**

`create_file.py`、`read_file.py`、`modify_file.py`、`delete_file.py`、`list_files.py` 对用户传入路径的处理不统一，部分工具仅做 `normalize`，部分工具使用字符串 `startswith` 判断目录归属，存在路径穿越和同前缀目录误判风险。

例如：

- `../outside.txt` 可能越出项目目录
- `../app-other/secret.txt` 可能因为字符串前缀相同而被误判为合法路径

**方案**

新增统一的路径防护工具 `python-agent/tools/path_guard.py`，通过 `realpath + commonpath` 的方式校验目标路径是否仍位于 `project_dir` 内部。

所有文件工具统一接入该能力，对非法路径直接抛出异常并拒绝执行。

**落地结果**

Python 工具链现在具备统一的目录边界校验能力，路径穿越和前缀绕过问题得到修复。

**涉及文件**

- `python-agent/tools/path_guard.py`
- `python-agent/tools/create_file.py`
- `python-agent/tools/read_file.py`
- `python-agent/tools/modify_file.py`
- `python-agent/tools/delete_file.py`
- `python-agent/tools/list_files.py`
- `python-agent/tests/test_path_guard_unittest.py`

## 3. 静态资源访问存在路径越界风险

**问题**

`StaticResourceController` 根据 `deployKey` 和通配路径直接拼接静态资源文件路径，缺少标准化后的目录归属校验，攻击者可通过构造路径尝试访问部署目录之外的文件。

此外，预览根目录原先与实际部署目录语义不一致，存在读取错误目录的风险。

**方案**

新增 Java 侧路径安全工具 `PathSafetyUtils`，统一使用 `Path.normalize()` 和根目录 containment 校验限制访问边界。

`StaticResourceController` 改为：

- 基于 `CODE_DEPLOY_ROOT_DIR` 读取静态资源
- 通过 `PathSafetyUtils.resolveInside(...)` 构建最终文件路径
- 对越界访问返回 `403 Forbidden`

**落地结果**

静态资源只允许从合法部署目录中读取，路径穿越尝试会被拒绝，预览目录与真实部署目录保持一致。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/utils/PathSafetyUtils.java`
- `src/main/java/com/rain/rainn0coding/controller/StaticResourceController.java`
- `src/test/java/com/rain/rainn0coding/utils/PathSafetyUtilsTest.java`

## 4. 排序字段直接透传，存在 SQL 注入风险

**问题**

`AppServiceImpl`、`ChatHistoryServiceImpl`、`UserServiceImpl` 在构造查询条件时，直接将外部传入的 `sortField` 交给 `orderBy`，缺少白名单约束，存在 SQL 注入面。

**方案**

新增 `SqlSafetyUtils.safeSortField(...)`，对排序字段做白名单过滤。各业务服务显式维护允许排序的字段集合，仅白名单字段允许进入排序语句，其余输入统一忽略或回退到默认排序。

**落地结果**

排序能力保留，同时关闭了任意字段拼接进入 SQL 的入口。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/utils/SqlSafetyUtils.java`
- `src/main/java/com/rain/rainn0coding/service/impl/AppServiceImpl.java`
- `src/main/java/com/rain/rainn0coding/service/impl/ChatHistoryServiceImpl.java`
- `src/main/java/com/rain/rainn0coding/service/impl/UserServiceImpl.java`
- `src/test/java/com/rain/rainn0coding/utils/SqlSafetyUtilsTest.java`

## 5. 管理端列表接口缺少分页上限

**问题**

`AppController`、`ChatHistoryController`、`UserController` 的管理端分页接口未限制 `pageSize`，`AppVersionController` 还存在无上限的 `list()` 查询。恶意或误用请求可能导致大页查询、数据库压力升高和接口响应异常。

**方案**

为管理端接口补充分页上限校验：

- App、ChatHistory、User 管理分页统一限制最大页大小
- AppVersion 的 `list()` 改为固定上限查询
- AppVersion 的 `page()` 接口补充分页参数校验

**落地结果**

管理端查询具备基础的资源保护能力，避免单次请求拉取过量数据。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/controller/AppController.java`
- `src/main/java/com/rain/rainn0coding/controller/ChatHistoryController.java`
- `src/main/java/com/rain/rainn0coding/controller/UserController.java`
- `src/main/java/com/rain/rainn0coding/controller/AppVersionController.java`

## 6. AppVersionController 缺少权限保护

**问题**

`AppVersionController` 原先对版本记录提供了直接的增删改查能力，但缺少管理员权限校验，存在敏感版本数据被未授权访问和修改的风险。

**方案**

为 `AppVersionController` 全部接口补充 `@AuthCheck(mustRole = UserConstant.ADMIN_ROLE)`，仅允许管理员访问版本数据。

**落地结果**

应用版本数据的读写权限被收敛到管理角色，安全边界更清晰。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/controller/AppVersionController.java`

## 7. 用户密码仍使用静态盐 MD5

**问题**

`UserServiceImpl` 原先通过“固定盐值 + MD5”保存密码，这属于明显过时的方案，抗暴力破解能力弱，也不利于长期安全治理。

**方案**

将密码哈希升级为 BCrypt，并保留对历史 MD5 密码的兼容校验：

- 新注册和新写入密码统一使用 BCrypt
- 老用户登录时若命中旧 MD5 校验，则在登录成功后自动迁移为 BCrypt

这样既能提升密码安全性，也避免一次性强制迁移带来的兼容问题。

**落地结果**

密码存储方案完成升级，系统进入“新密码强哈希、老密码渐进迁移”的状态。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/service/impl/UserServiceImpl.java`
- `src/test/java/com/rain/rainn0coding/service/impl/UserServiceImplTest.java`

## 8. App 列表转换仍存在 N+1 查询

**问题**

`AppServiceImpl.getAppVOList()` 虽然预先批量查询了用户，但在流式转换中仍调用 `getAppVO(app)`，而该方法内部再次通过 `userService.getById` 查询用户，导致 N+1 查询问题没有真正消除。

**方案**

在 `getAppVOList()` 中直接组装 `AppVO`，复用已批量查询到的用户映射，不再二次调用单条用户查询方法。

**落地结果**

App 列表转换过程去掉了重复查库逻辑，列表查询性能更稳定。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/service/impl/AppServiceImpl.java`

## 9. CORS 配置过宽，与凭证共享组合存在风险

**问题**

系统使用 Cookie 鉴权，但 CORS 配置允许带凭证的跨域访问时使用过宽的来源匹配，容易放大跨域滥用风险。

**方案**

将 CORS 允许来源改为通过配置项控制，默认仅放开本地开发常见来源，并保留通过环境变量扩展的能力。

同时将相关配置收敛到 `application.yml` 中，避免写死在代码中。

**落地结果**

CORS 从“默认过宽”调整为“默认收敛、按需放开”，更符合生产安全预期。

**涉及文件**

- `src/main/java/com/rain/rainn0coding/config/CorsConfig.java`
- `src/main/resources/application.yml`

## 10. 配置文件中存在硬编码敏感信息和本地产物泄漏风险

**问题**

项目中存在数据库、Redis、Python 服务地址等配置写死在主配置文件中的情况，Python Agent 的 `.env` 也包含真实密钥，不利于环境隔离和凭证管理。

同时，仓库缺少对 `.env`、本地数据库、日志、缓存目录等产物的忽略规则，容易把敏感或无关文件带入版本控制。

**方案**

完成两部分治理：

- `application.yml` 改为优先读取环境变量，保留安全默认值
- `python-agent/.env` 中的真实 key 替换为占位符
- `.gitignore` 补充本地产物、缓存文件、日志文件和敏感配置文件的忽略规则

**落地结果**

配置与密钥管理方式更符合多环境部署习惯，仓库污染和误提交风险同步下降。

**涉及文件**

- `src/main/resources/application.yml`
- `python-agent/.env`
- `.gitignore`

## 11. 回归测试被旧接口签名阻塞

**问题**

在补充 Java 单元测试后，`AiCodeGeneratorFacadeTest` 仍使用旧版 `generateAndSaveCodeStream` 方法签名，导致 Maven 在测试编译阶段报错，影响新增回归测试的执行。

**方案**

同步更新测试调用参数，使其与当前 `AiCodeGeneratorFacade` 的方法签名保持一致，仅修正编译阻塞，不改动原测试意图。

**落地结果**

Java 定向测试可以正常编译并执行，回归验证链路恢复。

**涉及文件**

- `src/test/java/com/rain/rainn0coding/core/AiCodeGeneratorFacadeTest.java`
