# AI 代码生成工作台前端骨架设计文档

日期：2026-06-30
项目：`yu-ai-code-mother`
范围：在已经删除旧前端的前提下，重新搭建一套全新的前端骨架，同时保留当前可用的登录页和注册页入口。

## 1. 目标

本次要做的不是“把所有功能一次做完”，而是先搭一套全新的前端骨架，让它一开始就有成熟 AI 产品的感觉，并且方便后续持续改页面、加页面、换布局、换风格。

这套骨架需要满足两件事：

- 现在就能作为新的产品外壳使用
- 以后你提出页面改动时，不需要每次都推倒重来

## 2. 产品定位

新的前端会被定义成一个“AI 应用生成工作台”，而不是传统后台管理系统，也不是纯展示型官网。

已经确认的方向如下：

- 首页类型：生成控制台型
- 整体气质：专业产品感
- 导航方式：左侧边栏
- 后续目标：方便持续调整页面，不锁死结构

## 3. 信息架构

第一版骨架先包含这些主要页面：

- `/login`：登录页
- `/register`：注册页
- `/`：生成控制台首页
- `/projects`：我的项目
- `/history`：历史记录
- `/templates`：模板 / 灵感
- `/settings`：账户设置

各页面职责如下：

### 3.1 登录页

用于账号密码登录，是未登录用户的主要入口。

### 3.2 注册页

用于新用户注册账号，表单结构与登录页保持统一风格。

### 3.3 生成控制台首页

这是登录后的默认首页，也是整套产品里最重要的页面。用户一进来就应该看到生成入口，而不是先看到一堆管理信息。

### 3.4 我的项目

用于查看所有已生成项目，后续可以逐步扩展出筛选、排序、状态分类等能力。

### 3.5 历史记录

用于查看历史生成会话、最近生成行为、后续的版本记录等。它和“我的项目”分开，避免首页信息太乱。

### 3.6 模板 / 灵感

用于放模板、示例项目、提示词方向、常用场景入口，帮助用户更快开始，而不是每次都从空白输入起步。

### 3.7 账户设置

用于放账号资料、偏好设置、安全设置等内容，后续可以继续拆细。

## 4. 路由原则

这次新骨架的路由有两个重要原则：

### 4.1 登录相关路由继续保留

`/login` 和 `/register` 必须继续可访问，并且保持当前能用。

### 4.2 旧前端路由不再暴露旧内容

以前旧前端的页面不能再被访问到。即使用户访问老路由，也不应该再看到原来的聊天页、管理页或旧界面。

## 5. 整体布局结构

整体采用“左侧边栏 + 右侧主内容区”的产品布局。

### 5.1 左侧边栏结构

左侧边栏分成 3 层：

- 顶部：品牌标识
- 中部：主导航
- 底部：用户信息和设置入口

具体内容建议如下：

- 品牌区
- `新建生成` 主按钮
- `生成控制台`
- `我的项目`
- `历史记录`
- `模板 / 灵感`
- `设置`
- 当前登录用户信息
- 退出登录

这样设计的目的，是让整套产品看起来更像成熟 SaaS，而不是临时拼出来的页面集合。

### 5.2 右侧主内容区结构

右侧内容区负责承载页面内容。每个页面都保持统一骨架：

- 页面标题
- 简短说明
- 主内容模块
- 辅助模块或状态模块

这样后续你改某个页面时，不需要把整个产品壳层一起改掉。

## 6. 首页设计

登录后的首页采用“生成控制台型”。

它的重点不是展示统计，也不是展示后台列表，而是让用户尽快开始生成代码。

### 6.1 首页模块优先级

首页按下面的优先级排列：

1. 主生成区
2. 最近继续区
3. 模板 / 灵感区
4. 系统反馈区

### 6.2 主生成区

这是首页的视觉中心。

这一块要包含：

- 大输入框
- 生成入口按钮
- 模式 / 类型选择壳层
- 必要的辅助操作

用户进入首页后，第一眼应该明确知道：“这里就是开始生成的地方。”

### 6.3 最近继续区

这一块用于帮助用户继续上一次的工作，而不是每次重新开始。

建议先放这些内容：

- 最近项目
- 最近一次会话
- 快速继续入口

### 6.4 模板 / 灵感区

这一块用于帮助用户更快开始。

可以先放：

- 常见生成方向
- 示例项目卡片
- 提示词分类入口

### 6.5 系统反馈区

这一块用于承接后续产品状态，例如：

- 生成中的状态
- 最近更新
- 错误提示
- 队列 / 排队提示

### 6.6 首页视觉定稿补充（2026-06-30）

首页首屏改为更接近对话式 RAG 工作台的布局，而不是传统控制台卡片拼接。

具体约束如下：

- 左侧为窄边栏，包含品牌区、`新建对话`、对话搜索、最近会话列表、底部功能跳转和当前用户信息
- 右侧主区以“超大标题 + 居中输入框”为绝对视觉中心，第一眼必须看到提问入口
- 输入框下方保留模式切换、发送按钮、快捷键提示和 3 个能力入口卡片
- 顶部导航做轻量化处理，只保留“新对话”语义和少量辅助跳转，不保留传统后台式大标题头部
- 整体风格统一为浅色 AI 工作台：暖白和浅蓝为主，辅以淡金高光、轻边框、圆角卡片和柔和阴影

这一补充只重做登录后的 `/` 首页，不改变 `/login`、`/register` 与其他二级页面的职责边界。

## 7. 各页面骨架说明

### 7.1 我的项目页

作用：

- 展示所有项目
- 提供后续扩展的页面容器
- 支持空状态

### 7.2 历史记录页

作用：

- 展示历史生成行为
- 为后续版本记录、时间线、调试轨迹预留位置

### 7.3 模板 / 灵感页

作用：

- 统一承接模板和灵感入口
- 强化产品的“快速开始”体验

### 7.4 设置页

作用：

- 承载账户相关信息
- 提供统一设置入口
- 后续可以继续拆成多个设置子模块

## 8. 组件边界

为了让你后面持续改页面，这套前端不能写成一个大页面，而是要分层拆开。

建议拆成以下几层：

- 布局层
- 页面层
- 模块层
- 基础组件层
- 请求服务层
- 状态层

### 8.1 布局层

负责：

- 左侧边栏
- 主内容容器
- 页面公共框架

不负责具体业务内容。

### 8.2 页面层

负责：

- 路由级页面
- 每个页面自己的标题、结构和模块组合

这样你以后重做某一页时，不会影响其他页。

### 8.3 模块层

负责：

- 主生成区
- 最近项目区
- 模板区
- 用户信息区
- 空状态区

这些模块会是你后面最常修改的地方，所以必须独立。

### 8.4 基础组件层

负责：

- 按钮
- 输入框
- 卡片
- 导航项
- 区块标题
- 空状态组件

这样以后你想整体换视觉风格，可以统一改，不用逐页改。

## 9. 状态管理

骨架阶段不需要上特别重的状态系统，但需要有基本的全局状态能力。

建议先管理这些状态：

- 当前登录用户
- 当前导航高亮状态
- 首页输入内容和生成模式
- 页面加载状态
- 全局提示消息

状态要够用，但不要在骨架阶段过度设计。

## 10. 请求层设计

接口请求逻辑要和页面展示分开，不能把登录、退出、获取用户这些逻辑直接散落在页面里。

第一阶段先统一处理这些请求：

- 登录
- 注册
- 获取当前用户
- 退出登录

后面项目、历史记录、模板等接口，也继续接到同一层里。

## 11. 视觉与体验规则

这次已经确认视觉方向是“专业产品感”，所以骨架实现时要遵守这些规则：

- 看起来像成熟产品，不像后台模板
- 首页强调“开始生成”
- 导航清晰稳定
- 信息层级明确
- 样式统一、克制、清爽
- 后续便于换内容和调整布局

这套界面要看起来像一个能继续长功能的 AI 产品，而不是一次性演示页。

## 12. 状态与异常处理

骨架阶段必须先把这些状态做好：

- 未登录状态
- 已登录但没有数据的空状态
- 加载中状态
- 接口失败状态
- 尚未实现页面的占位状态

预期行为如下：

- 未登录访问受保护页面时，自动回到登录页
- 接口失败时，不出现白屏
- 没数据时，也有清晰的空状态引导
- 暂时未完成的页面，也有统一壳层和说明

## 13. 验证目标

这套骨架完成后，至少要验证这些事情：

- 登录页仍然可用
- 注册页仍然可用
- 登录后能进入新的首页骨架
- 未登录访问受保护页面时会跳回登录页
- 左侧导航切换正常
- 旧前端页面不会再被看到
- 新静态资源没有语法错误
- `/api` 路径下仍然能正确提供前端页面

## 14. 本阶段不做的内容

这次骨架阶段暂时不包含以下内容：

- 完整项目 CRUD
- 真正的历史版本系统
- 真正的模板市场
- 复杂搜索和筛选逻辑
- 所有页面的最终细节打磨

这些内容后续都应该建立在这套骨架之上逐步补充，而不是现在一次塞满。

## 15. 推荐实现方式

推荐采用“产品骨架”方案。

原因如下：

- 符合你选择的“更像完整产品”
- 首页能直接体现 AI 产品核心价值
- 左侧边栏便于后续加页面
- 比“极简空壳”更像产品
- 比“重运营首页”更方便后续修改
